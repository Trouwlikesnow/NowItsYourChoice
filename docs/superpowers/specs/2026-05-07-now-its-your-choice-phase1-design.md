# NowItsYourChoice 阶段 1 设计文档：决策引擎 + 关注板块 MVP

**项目名**：NowItsYourChoice（缩写 NIYC）
**日期**：2026-05-07
**阶段**：Phase 1（MVP）
**预计周期**：2-3 周

---

## 1. 项目背景

构建一套个人投资管理系统 NowItsYourChoice，覆盖股票、基金、理财等多类资产的总体管理。系统核心包含：总资产看板、股票模块（看板/关注板块/复盘/决策引擎/知识库/外部内容引入）。

完整系统按五个阶段交付：

| 阶段 | 主题 | 周期 |
|---|---|---|
| 阶段 1 | 决策引擎 + 关注板块 MVP（本文档） | 2-3 周 |
| 阶段 2 | 总资产看板 + 持仓更新 | 1-2 周 |
| 阶段 3 | 外部内容引入（知乎/公众号/小红书） | 1 周 |
| 阶段 4 | 操作复盘体系 | 1-2 周 |
| 阶段 5 | 知识库系统化（持续运营） | 持续 |

阶段 1 选定为 MVP 的理由：决策引擎是用户高频核心需求，且打通 AkShare → 飞书 Bitable → Dify → 飞书 Bot 全链路后，后续模块都是在此架构上叠加。

---

## 2. 阶段 1 目标

### 范围内
- 用户可在飞书 Bot 输入股票代码，获得基于多周期技术指标 + 交易纪律 + 板块新闻的操作建议
- 系统每日自动抓取关注标的的日/周/月线行情、技术指标、板块新闻
- 关注板块和标的可在飞书 Bitable 维护，灵活增删
- 交易纪律可在 Dify Knowledge 维护，决策引擎自动召回
- 行情快照表可视化展示走势

### 不在范围内（推迟到后续阶段）
- 总资产看板（阶段 2）
- 持仓数据自动更新（阶段 2）
- 外部内容自动归档（阶段 3）
- 操作复盘记录与分析（阶段 4）
- 实时分钟级数据
- 直接对接券商 App 抓取持仓

## 3. 技术架构

### 3.1 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 数据源 | AkShare | A 股日/周/月线、板块、新闻；零积分免费。具体新闻接口（财联社 / 东财板块新闻 / 个股资讯）在实现阶段挑选 |
| 抓取与计算 | Python + pandas_ta | 在 GitHub Actions 上定时执行 |
| 调度 | GitHub Actions | 每日 15:30 触发；零运维 |

| 交互 | 飞书机器人（Bot） | 输入股票代码、推送建议 |

### 3.3 数据流向

**每日数据更新链路**（自动）

```
15:30 GitHub Actions 触发
  → 读 Bitable 标的表（拿到 50 只关注股票代码）
  → 对每只股票：
      ├─ AkShare 拉日/周/月 K 线（最近 250 日）
      ├─ pandas_ta 算指标（MA/MACD/RSI/KDJ/BOLL...）
      ├─ 写入指标表（删除旧行后写新行）
      └─ 写入行情快照表（写最新一天）
  → 对每个板块：
      └─ AkShare 拉板块新闻 → 写入板块新闻表
  → 滚动清理：行情快照超 90 天、新闻超 60 天的删除
  → 给飞书 Bot 推送"今日数据更新完成"
```

**决策请求链路**（交互）

```
用户在飞书 Bot 发送"$002594"
  → Bot 调 Dify 工作流 API（带 ticker_code 参数）
  → Dify 工作流：
      ├─ 节点1：从 Bitable 查标的表（拿基础信息）
      ├─ 节点2：从 Bitable 查指标表（该股票全周期全指标）
      ├─ 节点3：从 Bitable 查板块新闻表（该板块最近 7 天）
      ├─ 节点4：从 Knowledge 召回（交易纪律 + 知识）
      ├─ 节点5：拼装提示词 → 调 DeepSeek-V3
      └─ 节点6：把决策记录写回 Bitable 决策记录表
  → Dify 返回结果给 Bot
  → Bot 回复用户：操作建议 + 关键依据 + 风险提示
```

#### 交易纪律表（trading_rules）
| 字段 | 类型 | 说明 |
|---|---|---|
| 纪律标题 | 文本 | 简短表述 |
| 详细描述 | 多行文本 | |
| 类型 | 单选 | 仓位管理 / 进场 / 止损 / 止盈 / 心态 |
| 优先级 | 单选 | 必守 / 重要 / 参考 |
| 状态 | 单选 | 启用 / 停用 |

预计 30-100 行。

**注**：这张表里的纪律会**同步到 Dify Knowledge** 的 `knowledge_trading_rules` 库，让决策引擎能 RAG 召回。Bitable 是结构化管理入口（增删改），Dify Knowledge 是检索源。

阶段 1 同步方式：**手工同步**（用户在 Bitable 维护后，手动在 Dify Knowledge 上传 / 更新对应文档）。阶段 1 不实现自动同步以避免引入额外组件；阶段 5 知识库系统化时再考虑加自动同步脚本。

### 4.2 Dify Knowledge 内容组织

按"知识类型"建独立 Knowledge 库：

- **knowledge_trading_rules**：交易纪律（来自 Bitable 交易纪律表的同步）
- **knowledge_kol_views**：大V观点收藏（手工/阶段 3 自动写入）
- **knowledge_stock_basics**：股票投资基础知识（用户自行补充）

### 4.3 数据生命周期与清理

| 表 | 生命周期 | 清理策略 |
|---|---|---|
| 板块表 / 标的表 / 交易纪律表 | 永久 | 不清理 |
| 指标表 | 当前快照 | 每天先删该股票该周期所有行，再写新值 |
| 行情快照表 | 滚动 90 天 | 脚本删除 > 90 天的行 |
| 板块新闻表 | 滚动 60 天 | 脚本删除 > 60 天的行 |
| 决策记录表 | 永久（先不清理） | 阶段 4 复盘要用 |

## 5. 组件详细设计

### 5.1 GitHub Actions 数据更新脚本

**仓库结构**：

```
NowItsYourChoice/
├── .github/workflows/
│   └── daily.yml                  # 每日 15:30 触发
├── scripts/
│   ├── main.py                    # 入口
│   ├── data_fetcher.py            # AkShare 封装
│   ├── indicator_calc.py          # pandas_ta 计算（配置驱动）
│   ├── bitable_client.py          # 飞书 Bitable API 封装
│   ├── news_fetcher.py            # 板块新闻
│   └── config.py                  # 指标列表 / 周期配置
├── requirements.txt
└── README.md
```

**指标配置（config.py）示例**：

```python
INDICATOR_CONFIG = {
    "日": ["MA5", "MA10", "MA20", "MA60", "MACD", "RSI14", "KDJ", "BOLL"],
    "周": ["MA4", "MA13", "MA26", "MACD", "RSI14"],
    "月": ["MA6", "MA12", "MACD"],
}
```

**关键执行步骤**：

1. 从 Bitable 标的表拉关注股票列表
2. 对每只股票调 `stock_zh_a_hist(period='daily')` 拉 250 日数据
3. pandas_ta 算所有日级指标
4. 同样流程拉周线、月线，算各自指标
5. 批量写入指标表（先按 (股票, 周期) 删除旧行，再写新行）
6. 写入行情快照表（仅当日一行）
7. 更新标的表的当日字段（最新收盘价、涨跌幅等）
8. 板块新闻：调 AkShare 板块新闻接口，写入新闻表
9. 滚动清理过期数据
10. 完成后调用飞书 Bot Webhook 通知"今日数据已更新"或失败告警

**鲁棒性要求**：

- 单只股票数据失败不影响其他股票（try/except 围一只一只跑）
- AkShare 全局不可用时，发送告警但不让整个 Action 失败（标记为 warning）
- API 调用失败重试 3 次，指数退避
- 飞书 API 写入用 batch 接口，每批 ≤ 500 行

### 5.2 飞书 Bitable 配置

- 在飞书工作台新建一个"NowItsYourChoice"多维表格
- 创建以上 7 张数据表
- 创建至少以下视图（仪表盘）：
  - **关注标的**：标的表，按板块分组，显示当日涨跌幅、距高点回撤
  - **板块概览**：板块表，关联展示板块下标的表现

### 5.3 Dify 决策引擎工作流

**输入参数**：`ticker_code`（股票代码）、`user_question`（可选，用户的具体问题）

**工作流节点**：

1. **节点 1：参数校验与查找** — 校验股票代码格式；如果输入是股票名而非代码，调 AkShare 名称→代码转换；如果该代码不在标的表中，仍可继续走流程（临时拉数据），但在最终回复中标注"该标的未纳入日常关注，本次为临时分析"
2. **节点 2：拉标的基础信息** — HTTP 请求 Bitable Open API，查标的表 where 代码 = ticker_code
3. **节点 3：拉指标快照** — HTTP 请求 Bitable Open API，查指标表 where 代码 = ticker_code，返回该股票全周期全指标行（约 30 行）
4. **节点 4：拉板块新闻** — 根据节点 2 返回的板块名，查板块新闻表最近 7 天的新闻
5. **节点 5：召回交易纪律** — Knowledge 检索（query：股票名 + 板块 + "交易纪律"）
6. **节点 6：召回相关知识** — Knowledge 检索（query：板块名 + 用户问题）
7. **节点 7：拼装提示词** — 把以上信息按模板组装成结构化文本
8. **节点 8：调用 DeepSeek-V3** — 大模型节点，输出操作建议（结构化）
9. **节点 9：解析输出** — 把模型输出拆成"建议/依据/风险"三段
10. **节点 10：写回决策记录** — HTTP 请求 Bitable Open API，写入决策记录表
11. **输出**：返回给飞书 Bot 的回复内容

**Prompt 框架（节点 7 模板）**：

```
你是一位严谨的 A 股投资分析助手。请基于以下信息，给出对【{股票名}({代码})】的操作建议。

# 标的基础信息
板块：{板块}
当前持有：{持有量}（成本价：{成本价}）
最新收盘价：{价格}（涨跌幅 {涨跌幅}%）
60 日高点：{高点}，距高点回撤：{回撤}%

# 多周期技术指标
## 日线
{指标列表，含数值和信号状态}

## 周线
{指标列表}

## 月线
{指标列表}

# 板块近期新闻（最近 7 天）
{新闻标题列表}

# 适用的交易纪律
{从 Knowledge 召回的相关纪律}

# 相关投资知识
{从 Knowledge 召回的相关内容}

# 用户问题
{user_question 或"该如何操作？"}

请按以下格式输出：

## 操作建议
（明确的操作动作，例如：建议轻仓买入 / 持有观望 / 减仓一半 / 卖出）

## 关键依据
（3-5 条核心理由，结合多周期指标和板块新闻）

## 风险提示
（2-3 条主要风险点）

## 适用纪律
（本次决策遵循/触发的纪律条款）

要求：保持中立审慎，不夸张表述；明确指出关键数据；如果信息不充分，明确说明并建议补充什么。
```

### 5.4 飞书 Bot 接入

**Bot 能力**：

1. **基础对话**：用户发送 `$002594` 或 `比亚迪` 触发决策引擎
   - 解析输入：股票代码或股票名（名 → 代码用 AkShare 查表）
   - 调 Dify 工作流 API，等待结果
   - 把结果格式化成飞书消息卡片回复

2. **每日通知**：每日 15:30 数据更新完成后推送
   - 内容：今日关注标的的涨跌排行 TOP 5 / 触发指标信号的标的提醒

3. **指令**（阶段 1 实现）：
   - `$代码` 或 `名称`：决策建议
   - `/list`：列出当前关注标的
   - `/sector 板块名`：列出该板块所有标的当日表现
   - `/help`：帮助

**实现方式**：

- 在飞书开放平台创建自建应用，开启"机器人"能力
- 配置事件订阅 → 接收消息事件
- 后端服务接收事件，分发到对应处理器
- 后端服务部署：因为已有 Dify Cloud，可以用 Dify 自带的 API 暴露能力 + 飞书 Bot 直接调 Dify API（最简）；或者写一个轻量 webhook 服务部署到 Cloudflare Workers / Vercel

**推荐方式**：飞书 Bot → Dify API（直连，无中间服务）。Dify 工作流入口直接接收飞书 Webhook payload，工作流内部解析、处理、回复。

## 7. 设计反模式（明确避免）

1. **不要在 Bitable 用大量跨表 lookup 公式**：每次刷新都扫全表，性能杀手。聚合数据靠 Python 脚本预计算后写入。
2. **不要把原始 OHLCV 全历史存 Bitable**：AkShare 是源头，需要时再取，本系统只缓存"决策需要的精炼数据 + 滚动窗口快照"。
3. **不要把数据 commit 进 GitHub repo**：repo 会膨胀，commit 历史失去意义。
4. **不要在飞书 Bitable 公式字段做大范围聚合**：每次打开都重算。
5. **不要在 Dify 工作流里硬编码股票代码或板块名**：所有配置走 Bitable，工作流参数化。
6. **不要在指标表用宽表（每个指标一列）**：扩展新指标要改表。坚持长格式。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| AkShare 接口偶发不稳定 | 当日数据缺失 | 单股失败不影响整体；预留 BaoStock 切换钩子供后期 |
| 飞书 Open API 限流 | 写入超时 | 批量写、指数退避重试 |
| Dify Cloud 免费层不够用 | 决策响应失败 | 先上线监控用量，必要时升级或自部署（参考 8GB Mac mini 评估） |
| GitHub Actions 队列延迟 | 数据延迟分钟级 | 对每日级别决策可接受 |
| DeepSeek-V3 输出格式不稳定 | 解析失败 | Prompt 强约束 + 解析失败时原样返回 |
